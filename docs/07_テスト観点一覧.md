## 2. テスト分類

| 区分 | 目的 |
| --- | --- |
| API 契約テスト | request / response 形式とステータスコードの保証 |
| バックエンド単体テスト | サービス層 / 例外 / バリデーションの保証 |
| フロントエンド表示テスト | UI 状態遷移、表示内容の保証 |
| 結合テスト | モード選択から結果表示までの連携確認 |
| ローカル連携テスト | Ollama / NDLOCR-Lite 接続時の挙動確認 |
| 受け入れテスト | 利用者視点での操作完結性確認 |

## 3. Phase 1 テスト観点

## 3.1 API 契約

### 正常系

- `POST /api/summarize` が `multipart/form-data` を受け付ける
- `file` + `ocr_mode` + `summary_mode` を送信できる
- `api` OCR + `api` 要約で 200 を返す
- レスポンスに `ocrMode` / `summaryMode` が含まれる
- `warnings` が常に配列で返る

### 異常系

- `file` 未指定で 400
- 空ファイルで 400
- 非対応形式で 400
- サイズ超過で 413
- `ocr_mode` 不正で 400
- `summary_mode` 不正で 400

## 3.2 バックエンド単体

### モード値検証

- 許可 OCR モードのみ受け付ける
- 許可要約モードのみ受け付ける

### 既存経路維持

- `api` OCR が既存の OpenAI 抽出経路を通る
- `api` 要約が既存の OpenAI 要約経路を通る

### 暫定未接続エラー

- `ocr_mode=local_llm` で `LOCAL_LLM_UNAVAILABLE`
- `ocr_mode=high_accuracy` で `NDLOCR_UNAVAILABLE`
- `summary_mode=local_llm` で `LOCAL_LLM_UNAVAILABLE`

## 3.3 フロントエンド表示

### 初期表示

- タイトルが表示される
- OCR 方式ラジオボタンが表示される
- 要約方式ラジオボタンが表示される
- 初期値が `api` / `api`

### 入力操作

- JPG / PNG を選択できる
- 非対応形式を選ぶと警告表示される
- ファイル件数が表示される
- クリア操作でファイル選択が初期化される

### ボタン状態

- 有効ファイルがないと `要約実行` は無効
- 実行中は `要約実行` が再押下できない
- 実行中はモード変更を無効化できる

## 3.4 結果表示

### 単一画像

- 文書種別が表示される
- `ocrMode` / `summaryMode` が表示される
- 要約文が表示される
- 抽出テキストを折りたたみ表示できる

### 複数画像

- 総件数 / 成功件数 / 失敗件数が表示される
- 各ファイル結果がカード表示される
- 失敗ファイルはエラーカードとして表示される

## 4. Phase 2 テスト観点

## 4.1 Ollama OCR

### 正常系

- `ocr_mode=local_llm` で OCR が成功する
- 日本語テキストを返せる
- 画像入力が想定形式で受理される

### 異常系

- Ollama 未起動で `LOCAL_LLM_UNAVAILABLE`
- モデル未取得で明示エラー
- タイムアウト時に適切なエラー

## 4.2 Ollama 要約

### 正常系

- `summary_mode=local_llm` で要約が成功する
- `structuredData` 互換形式に変換できる

### 異常系

- 応答が空で `SUMMARY_GENERATION_FAILED`
- 構造化形式に整形できない場合のハンドリング

## 4.3 NDLOCR-Lite

### 正常系

- `ocr_mode=high_accuracy` で OCR が成功する
- OCR 結果が既存正規化層に流れる
- 帳票 / 紙文書サンプルで期待どおりのテキストが得られる

### 異常系

- 実行ファイル / モジュールが見つからない
- CLI 実行失敗
- タイムアウト
- OCR 結果が空

## 5. Phase 3 テスト観点

## 5.1 推奨モード表示

- 入力や文脈に応じた推奨が表示される
- 推奨表示がモード選択を妨げない

## 5.2 高精度後処理改善

- `NDLOCR-Lite` 後処理によって明らかな崩れが改善される
- 改善処理が誤った補完を増やしすぎない

## 5.3 warnings 表示

- warnings が視認できる
- 結果表示の主文を邪魔しない

## 6. モード組み合わせ別観点

| OCR | 要約 | 主観点 |
| --- | --- | --- |
| `api` | `api` | 後方互換維持 |
| `api` | `local_llm` | API OCR とローカル要約の接続整合 |
| `local_llm` | `api` | ローカル OCR と API 要約の接続整合 |
| `local_llm` | `local_llm` | ローカル完結動作 |
| `high_accuracy` | `api` | 文書向け OCR と既存要約の接続整合 |
| `high_accuracy` | `local_llm` | 高精度 OCR とローカル要約の接続整合 |

## 7. 受け入れテスト観点

### 利用者視点の確認項目

- 利用者が方式を迷わず選べる
- 方式を選んだままファイル送信できる
- 成功時に「何が書いてある画像だったか」が分かる
- 失敗時に「何を直せばよいか」が分かる
- ローカル未設定時に原因が伝わる

### 手動確認シナリオ

1. `api` / `api` で給与明細画像を送る
2. `api` / `local_llm` を選択し、未設定エラーを確認する
3. `high_accuracy` / `api` を選択し、未設定エラーまたは成功を確認する
4. 複数ファイルを送って、成功 / 失敗混在時の見え方を確認する

## 8. 優先度

### 最優先

- Phase 1 API 契約テスト
- `api` / `api` 後方互換テスト
- UI のモード選択表示テスト

### 高

- 方式依存エラーのテスト
- 単一 / 複数結果表示テスト

### 中

- ローカル連携正常系テスト
- warnings 表示改善テスト
