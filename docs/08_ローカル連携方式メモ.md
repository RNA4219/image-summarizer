# ローカル連携方式メモ v0.1

## 1. 目的

本書は、`Ollama` および `NDLOCR-Lite` を本アプリへ統合する際の接続方式候補、
前提条件、メリット / リスク、推奨方針を整理するための設計メモである。

本書は実装そのものではなく、Phase 2 着手前の判断材料として扱う。

## 2. 対象

本書の対象は以下の 2 系統である。

1. `Ollama` を利用したローカル LLM OCR / 要約
2. `NDLOCR-Lite` を利用した高精度 OCR

## 3. Ollama 連携

## 3.1 連携対象

- `ocr_mode=local_llm`
- `summary_mode=local_llm`

## 3.2 想定構成

```text
Frontend
  -> FastAPI Backend
      -> Ollama HTTP API
          -> Local multimodal / text model
```

## 3.3 接続方式候補

### 方式 A: Ollama HTTP API 直接呼び出し

#### 概要

- FastAPI から Ollama の HTTP API を直接呼び出す

#### メリット

- 実装がシンプル
- 依存が少ない
- ログやタイムアウト制御をバックエンド側で持ちやすい

#### デメリット

- モデルごとの差分吸収を自前で行う必要がある
- 画像入力形式の差異を吸収する必要がある

#### 推奨度

- 高

### 方式 B: ラッパー SDK 経由

#### 概要

- Ollama クライアントライブラリやラッパー経由で呼び出す

#### メリット

- 呼び出しコードが短くなる可能性がある

#### デメリット

- SDK 依存が増える
- 画像対応や応答形式が想定と合わない可能性がある

#### 推奨度

- 中

## 3.4 OCR 用 Ollama 連携方針

### 入力

- 前処理済み画像バイト列
- content type
- 補助プロンプト

### 出力

- 生の抽出テキスト

### 前提

- マルチモーダル入力に対応したモデルが必要
- 日本語 OCR で一定品質を出せるモデル選定が必要

### リスク

- モデルによって OCR 精度差が大きい
- レスポンスが冗長になりやすい
- 応答安定性がクラウド API より弱い可能性がある

## 3.5 要約用 Ollama 連携方針

### 入力

- 正規化済みテキスト
- 要約 / 構造化出力用プロンプト

### 出力

- できれば `structuredData` 相当の JSON
- 難しい場合は自然文要約 + 後段整形

### リスク

- Structured Output の安定性が OpenAI API より弱い可能性がある
- JSON 崩れ対策が必要

### 推奨

- Phase 2 初期は「自然文 + 簡易構造化」で始める
- 厳密 JSON は段階的に強化する

## 4. NDLOCR-Lite 連携

## 4.1 対象

- `ocr_mode=high_accuracy`

## 4.2 想定構成

```text
Frontend
  -> FastAPI Backend
      -> NDLOCR-Lite
          -> OCR text
      -> normalization
      -> summary
```

## 4.3 接続方式候補

### 方式 A: CLI 実行

#### 概要

- FastAPI から `ndlocr-lite` CLI を呼び出す
- 一時ファイル経由で入力画像と出力結果を受け渡す

#### メリット

- 元 OSS との疎結合を保ちやすい
- upstream 更新に追従しやすい
- 既存の CLI 使用例に沿って導入しやすい

#### デメリット

- 一時ファイル管理が必要
- 実行時間とタイムアウト制御が必要
- 実行環境依存が増える

#### 推奨度

- 高

### 方式 B: Python モジュール import

#### 概要

- `ndlocr-lite` の内部モジュールを直接 import して利用する

#### メリット

- プロセス起動コストを抑えられる可能性がある
- 中間ファイルを減らせる可能性がある

#### デメリット

- upstream の内部構造変更に弱い
- 依存解決や import パス管理が複雑になる

#### 推奨度

- 中

## 4.4 推奨初期方針

- Phase 2 初期は CLI 実行で開始する
- まずは「安定して OCR 結果を取れること」を優先する
- 性能や保守性に問題が出た場合のみ import 方式を再検討する

## 4.5 入出力方針

### 入力

- 単一画像ファイル
- 一時ディレクトリ

### 出力

- OCR 結果テキスト
- 必要に応じて可視化画像や補助ファイル

### 既存アプリへの接続

- `NDLOCR-Lite` で得た OCR テキストを既存 `normalization` に渡す
- その後の要約経路は `api` / `local_llm` どちらも再利用する

## 5. 推奨実装順

### 5.1 Ollama

1. 接続先 URL とモデル名を設定値化する
2. ヘルスチェック相当の疎通確認を作る
3. 要約の `local_llm` を先に実装する
4. OCR の `local_llm` を実装する

### 5.2 NDLOCR-Lite

1. CLI 実行の疎通確認を行う
2. 一時ファイル入出力方式を決める
3. OCR 結果文字列を取得する
4. 既存 `normalization` に接続する

## 6. 設定項目候補

### 6.1 Ollama

- `OLLAMA_BASE_URL`
- `OLLAMA_OCR_MODEL`
- `OLLAMA_SUMMARY_MODEL`
- `OLLAMA_TIMEOUT_SECONDS`

推奨既定モデル:

- `Qwen3.6-35B-A3B`

### 6.2 NDLOCR-Lite

- `NDLOCR_LITE_PATH`
- `NDLOCR_TIMEOUT_SECONDS`
- `NDLOCR_TEMP_DIR`

## 7. リスクと対策

| リスク | 内容 | 対策案 |
| --- | --- | --- |
| モデル品質差 | Ollama モデルごとの差異が大きい | モデル候補を限定する |
| JSON 崩れ | ローカル要約の構造化が不安定 | 後段で整形 / バリデーションする |
| 実行時間増加 | ローカル処理が遅い | タイムアウト設定、UI で案内する |
| CLI 失敗 | NDLOCR-Lite 実行環境差異 | 明示エラーと RUNBOOK で補う |
| upstream 変更 | NDLOCR-Lite 内部構造が変わる | 初期は CLI 連携に留める |

## 8. 推奨結論

現時点での推奨は以下のとおり。

- Ollama は HTTP API 直接呼び出しで始める
- ローカル要約を先に実装し、その後 OCR を実装する
- `NDLOCR-Lite` は CLI 連携で始める
- `high_accuracy` モードは `NDLOCR-Lite` 単独 OCR + 既存正規化 / 要約で開始する
- 複数 OCR 結果の統合は後続フェーズで再検討する
- 自動フォールバックは入れず、失敗理由をエラーコードで返す

## 8.1 Phase 2 で固定する事項

- `ocr_mode=local_llm` は Ollama マルチモーダル OCR とする
- `summary_mode=local_llm` は Ollama による整形 / 要約とする
- `ocr_mode=high_accuracy` は `NDLOCR-Lite` 主 OCR とする
- `NDLOCR-Lite` は CLI 実行方式を正式採用とする
- ローカル依存未整備時は API サーバ起動ではなくリクエスト実行時に失敗させる
- Ollama の推奨既定モデルは `Qwen3.6-35B-A3B` とする
- `NDLOCR-Lite` の単一画像実行は `ndlocr-lite --sourceimg <path> --output <dir>` を正式仕様とする
- OCR 本文は `NDLOCR-Lite` 出力の `<stem>.txt` を一次採用とする
