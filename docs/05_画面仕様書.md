# 画面仕様書 v0.1

## 1. 目的

本書は、[01_要件定義.md](./01_要件定義.md)、
[02_仕様書.md](./02_仕様書.md)、
[04_APIインターフェース定義.md](./04_APIインターフェース定義.md)
をもとに、フロントエンド画面の構成、表示項目、状態遷移、文言方針を定義する。

本書は、実装前に以下を固定することを目的とする。

- ユーザーが見る画面構成
- OCR / 要約方式の選択 UI
- 実行中 / 成功 / 失敗時の見え方
- 単一画像 / 複数画像時の結果表示
- 各モードの補助説明文

## 2. 画面一覧

本アプリは MVP 時点では 1 画面構成とする。

| 画面ID | 画面名 | 用途 |
| --- | --- | --- |
| `SCR-01` | 画像要約画面 | 画像アップロード、方式選択、要約実行、結果確認 |

補助的に、バックエンド側には `/docs` を利用した API ドキュメント画面が存在するが、
本画面仕様書の主対象はユーザー向けフロント画面とする。

## 3. 画面全体構成

`SCR-01` は以下の 4 ブロックで構成する。

1. ヘッダ領域
2. 入力 / 設定領域
3. 実行ステータス / メッセージ領域
4. 結果表示領域

### 3.1 レイアウト方針

- PC では縦並びの 1 カラムを基本とする
- モバイルでも同一順序を維持する
- モード選択はアップロードボタンより上または直下に配置し、実行前に必ず目に入るようにする
- 成功 / 失敗結果は入力ブロックの下に表示する
- 青と白を基調にし、カードベースで整理されたモダンな見た目にする

### 3.2 ビジュアル方針

- 主配色は青と白とする
- 背景は白またはごく薄い青系の明色を使う
- 主操作ボタン、アクティブ状態、リンク、強調要素は青系で統一する
- 情報カードは白ベースにし、境界線や影は控えめに使う
- 全体として清潔感、信頼感、可読性を優先する
- 多色構成や過度に装飾的な演出は避ける

## 4. `SCR-01` 画像要約画面

## 4.1 ヘッダ領域

### 表示項目

| 項目 | 種別 | 必須 | 内容 |
| --- | --- | --- | --- |
| タイトル | テキスト | 必須 | `画像要約アプリ` |
| 説明文 | テキスト | 必須 | 対応形式、用途、処理概要を短く案内する |

#### 見た目方針

- タイトルは青系アクセントで視認性を持たせる
- 説明文は白背景上で読みやすい濃い文字色を用いる

### 表示文言例

```text
JPG / PNG ファイルをアップロードしてください。
給与明細や帳票などの画像からテキストを抽出し、要約します。
```

## 4.2 入力 / 設定領域

入力 / 設定領域は以下の順序で表示する。

1. OCR 方式選択
2. 要約方式選択
3. ファイル選択
4. 選択ファイル一覧
5. 実行ボタン / クリアボタン

### 4.2.1 OCR 方式選択

#### UI 種別

- ラジオボタン

#### 選択肢

| 値 | 表示ラベル | 説明 |
| --- | --- | --- |
| `api` | `標準 (API OCR)` | 汎用性が高く、スクリーンショットや半構造文書向け |
| `local_llm` | `ローカル (Ollama OCR)` | ローカル完結を優先する場合に利用 |
| `high_accuracy` | `高精度 (NDLOCR-Lite)` | 帳票・紙文書向けの精度向上を狙う |

#### 初期値

- `api`

#### 補助文言

- `標準 (API OCR)`: まずはこちらを推奨します
- `ローカル (Ollama OCR)`: ローカル環境設定が必要です
- `高精度 (NDLOCR-Lite)`: 文書画像向けです

### 4.2.2 要約方式選択

#### UI 種別

- ラジオボタン

#### 選択肢

| 値 | 表示ラベル | 説明 |
| --- | --- | --- |
| `api` | `クラウド要約 (API)` | 既存の標準要約経路 |
| `local_llm` | `ローカル要約 (Ollama)` | ローカル完結を優先する場合に利用 |

#### 初期値

- `api`

#### 補助文言

- `クラウド要約 (API)`: 既定の推奨方式
- `ローカル要約 (Ollama)`: ローカル環境設定が必要です

### 4.2.3 ファイル選択

#### UI 種別

- ファイル入力
- ボタン押下で OS のファイル選択ダイアログを開く

#### 制約

- 複数選択可
- 許可拡張子は JPG / JPEG / PNG
- 非対応形式が選択された場合、エラー表示を行う

### 4.2.4 選択ファイル一覧

#### 表示内容

- 選択件数
- ファイル名一覧
- 各ファイルの削除操作
- 非対応形式ファイルはエラー色で表示

#### 表示例

```text
選択ファイル: 2件
- payslip_01.png
- payslip_02.jpg
```

### 4.2.5 操作ボタン

| ボタン | 表示名 | 動作 |
| --- | --- | --- |
| 主操作 | `要約実行` | 現在の設定で API 実行 |
| 補助操作 | `クリア` | 選択ファイルと一部状態を初期化 |

#### ボタン活性条件

- `要約実行` は有効なファイルが 1 件以上あり、かつ実行中でない場合に活性
- `クリア` はファイル選択がある場合に表示または活性

#### 見た目方針

- `要約実行` は青系の主ボタンとする
- `クリア` は白背景に青系アクセントの副ボタンとする

## 4.3 実行ステータス / メッセージ領域

### 4.3.1 ローディング表示

#### 表示条件

- 実行中

#### 表示内容

- スピナー
- 実行中であることを示す視覚表現
- 二重送信防止のため操作 UI を無効化

#### 文言例

```text
処理中です...
```

### 4.3.2 情報メッセージ

#### 表示条件

- `local_llm` または `high_accuracy` を選択している場合

#### 目的

- ローカル連携に環境設定が必要であることを事前に知らせる

#### 文言例

```text
ローカル系モードは追加設定が必要です。未設定の場合は方式依存エラーが表示されます。
```

### 4.3.3 エラーメッセージ

#### 表示位置

- 入力 / 設定領域の直下

#### 表示方針

- 1 回の実行につき主要エラー 1 件を見せる
- ユーザー向けに簡潔な日本語で表示する
- 技術詳細は出しすぎない

#### エラー文言例

- `非対応のファイル形式が含まれています。JPG / PNG のみ選択してください。`
- `ローカル LLM に接続できません。Ollama の設定を確認してください。`
- `NDLOCR-Lite を利用できません。ローカル設定を確認してください。`

## 4.4 結果表示領域

結果表示領域は、単一画像時と複数画像時で見せ方を分ける。

### 4.4.1 単一画像結果表示

#### 表示項目

| 項目 | 必須 | 説明 |
| --- | --- | --- |
| 文書種別 | 推奨 | `structuredData.documentType` を表示 |
| OCR 方式 | 必須 | `ocrMode` を表示 |
| 要約方式 | 必須 | `summaryMode` を表示 |
| 対象期間 | 任意 | `structuredData.targetPeriod` がある場合のみ表示 |
| ファイル名 | 必須 | 元ファイル名 |
| 要約文 | 必須 | 主結果として表示 |
| 詳細情報 | 任意 | 折りたたみ表示 |
| 不確実項目 | 任意 | 注意表示 |
| 抽出テキスト | 任意 | 折りたたみ表示 |
| warnings | 任意 | 必要に応じて表示 |

#### 表示優先度

1. 要約文
2. 文書種別 / 対象期間 / 利用モード
3. 詳細情報
4. 不確実項目
5. 抽出テキスト

### 4.4.2 複数画像結果表示

#### 表示項目

| 項目 | 必須 | 説明 |
| --- | --- | --- |
| 総件数 | 必須 | `totalFiles` |
| 成功件数 | 必須 | `successCount` |
| 失敗件数 | 必須 | `errorCount` |
| 各ファイル結果 | 必須 | ファイルごとのカード表示 |

#### 各カード表示項目

- ファイル名
- OCR 方式
- 要約方式
- 成功時は要約結果
- 失敗時はエラーメッセージ

### 4.4.3 成功カードの見え方

- 左アクセント色を成功色にする
- 文書種別を見出しに置く
- `OCR: xxx`、`要約: xxx` をチップ表示する
- 詳細情報、抽出テキストは折りたたみ式にする
- カード本体は白ベースとし、青系アクセントで全体の一貫性を保つ

### 4.4.4 失敗カードの見え方

- 左アクセント色をエラー色にする
- ファイル名を見出しに置く
- エラーメッセージを本文に表示する

## 5. 状態遷移

## 5.1 画面状態

| 状態 | 説明 |
| --- | --- |
| `idle` | 初期状態 |
| `loading` | 実行中 |
| `success-single` | 単一ファイル成功 |
| `success-multi` | 複数ファイル結果表示 |
| `error` | 実行失敗 |

## 5.2 遷移

```mermaid
stateDiagram-v2
  [*] --> idle
  idle --> loading: 要約実行
  loading --> success-single: 単一画像成功
  loading --> success-multi: 複数画像処理完了
  loading --> error: API失敗/設定不備/入力エラー
  error --> loading: 再実行
  success-single --> loading: 再実行
  success-multi --> loading: 再実行
```

## 6. 表示文言方針

### 6.1 トーン

- 簡潔
- 日本語で自然
- 原因と次の行動が分かる
- 見た目は青と白を基調に、モダンで落ち着いた雰囲気にする

### 6.2 文言原則

- エラーは利用者向けに説明する
- 技術用語は必要最小限にする
- モード名は UI ラベルと整合させる

### 6.3 用語統一

| 用語 | 画面での表記 |
| --- | --- |
| API OCR | `標準 (API OCR)` |
| local_llm OCR | `ローカル (Ollama OCR)` |
| high_accuracy OCR | `高精度 (NDLOCR-Lite)` |
| API summary | `クラウド要約 (API)` |
| local_llm summary | `ローカル要約 (Ollama)` |

## 7. 入力チェック UI 方針

### 7.1 送信前チェック

- 有効ファイルが 0 件なら `要約実行` を無効化
- 非対応形式を含む場合、警告表示を出す
- 実行中はファイル追加、削除、モード変更を無効化する

### 7.2 実行後エラー

- サーバーから返った `error.message` をそのまま利用者向けに表示する
- 複数ファイル時は全体失敗ではなく個別失敗をカードに反映できる設計とする

## 8. アクセシビリティ / レスポンシブ方針

- ラジオボタンはラベルを明確に関連付ける
- 色だけで状態を伝えず、アイコンや文言も併用する
- 折りたたみ要素はキーボード操作可能とする
- モバイルではラジオボタンを縦並びにしてもよい
- 結果カードは横スクロールに依存しない構成を優先する
