# API詳細設計 v0.2

## 1. Overview

本仕様は、[01_要件定義.md](./01_要件定義.md) で定義した
「OCR 方式」と「要約方式」の分離に対応する API / 内部設計を定める。

本アプリの処理は以下の 3 段階で構成する。

1. OCR / テキスト抽出
2. テキスト整形
3. 要約生成

OCR と要約は独立に切り替え可能とし、利用者は用途に応じて処理方式を選択できる。

## 2. API一覧

MVP の必須 API は `POST /api/summarize` とする。
実装では拡張機能として `POST /api/summarize-multiple` を維持してよいが、
正式仕様の主対象は単一画像処理とする。

| メソッド | パス | 用途 | 優先度 |
| --- | --- | --- | --- |
| POST | `/api/summarize` | 単一画像に対して、選択された OCR / 要約方式で処理を実行する | 必須 |
| POST | `/api/summarize-multiple` | 複数画像をまとめて処理する | 拡張 |
| GET | `/health` | ローカル確認用ヘルスチェック | 任意 |

## 3. Modes

## 3.1 OCR モード

| モード値 | 表示名 | 説明 |
| --- | --- | --- |
| `api` | API OCR | OpenAI API を利用したクラウド OCR |
| `local_llm` | ローカル LLM OCR | Ollama のマルチモーダル LLM を利用した OCR |
| `high_accuracy` | 高精度 OCR | `NDLOCR-Lite` を利用した文書向け OCR |

## 3.2 要約モード

| モード値 | 表示名 | 説明 |
| --- | --- | --- |
| `api` | API 要約 | OpenAI API を利用した整形 / 要約 |
| `local_llm` | ローカル LLM 要約 | Ollama のローカル LLM を利用した整形 / 要約 |

## 3.3 サポートする組み合わせ

- `api` OCR + `api` 要約
- `api` OCR + `local_llm` 要約
- `local_llm` OCR + `api` 要約
- `local_llm` OCR + `local_llm` 要約
- `high_accuracy` OCR + `api` 要約
- `high_accuracy` OCR + `local_llm` 要約

初期段階では、`high_accuracy` OCR は `NDLOCR-Lite` を主処理とし、
複数 OCR 結果の高度なアンサンブル統合は行わない。

## 4. Endpoint Specification

## 4.1 POST `/api/summarize`

### 目的

アップロードされた単一画像に対して、
選択された OCR モードと要約モードに従って処理を実行し、結果を返す。

### リクエスト

- **Method**: `POST`
- **Path**: `/api/summarize`
- **Content-Type**: `multipart/form-data`

### フォーム項目

| フィールド名 | 型 | 必須 | 説明 |
| --- | --- | --- | --- |
| `file` | binary | 必須 | JPG / JPEG / PNG の画像ファイル |
| `ocr_mode` | string | 必須 | `api` / `local_llm` / `high_accuracy` |
| `summary_mode` | string | 必須 | `api` / `local_llm` |

### 受け入れ条件

- 対応形式: `image/jpeg`, `image/png`
- ファイルサイズ上限: 10MB
- 0 byte ファイルは不可
- 1 リクエスト 1 ファイルのみ
- `ocr_mode` は定義済み値のみ受け付ける
- `summary_mode` は定義済み値のみ受け付ける

### FastAPI 想定シグネチャ

```python
@router.post("/api/summarize", response_model=SummarizeResponse)
async def summarize(
    file: UploadFile = File(...),
    ocr_mode: str = Form(...),
    summary_mode: str = Form(...),
) -> SummarizeResponse:
    ...
```

## 4.2 POST `/api/summarize-multiple`

### 目的

複数画像に対して同一の OCR モード / 要約モードを適用し、
ファイルごとの結果一覧を返す。

### リクエスト

- **Method**: `POST`
- **Path**: `/api/summarize-multiple`
- **Content-Type**: `multipart/form-data`

### フォーム項目

| フィールド名 | 型 | 必須 | 説明 |
| --- | --- | --- | --- |
| `files` | binary[] | 必須 | 複数画像ファイル |
| `ocr_mode` | string | 必須 | `api` / `local_llm` / `high_accuracy` |
| `summary_mode` | string | 必須 | `api` / `local_llm` |

## 6. Processing Flow

## 6.1 正常系

1. フロントから画像ファイル、`ocr_mode`、`summary_mode` を送信する
2. API がファイル形式 / サイズ / 空ファイル / モード値を検証する
3. 画像バイト列を取得する
4. 選択された OCR モードに応じて抽出処理を実行する
5. 正規化 / 整形処理を実行する
6. 選択された要約モードに応じて要約処理を実行する
7. 結果を JSON として返却する

## 6.2 異常系

- ファイル未添付
- 非対応形式
- サイズ超過
- OCR モード不正
- 要約モード不正
- OpenAI API 呼び出し失敗
- Ollama 未起動またはモデル未準備
- `NDLOCR-Lite` 実行失敗
- 抽出結果が空
- 要約生成失敗

## 7. Status Codes and Error Codes

| ステータス | code | 意味 |
| --- | --- | --- |
| 200 | - | 正常終了 |
| 400 | `FILE_REQUIRED` | ファイル未指定 |
| 400 | `EMPTY_FILE` | 空ファイル |
| 400 | `UNSUPPORTED_FILE_TYPE` | 非対応形式 |
| 400 | `INVALID_OCR_MODE` | OCR モード不正 |
| 400 | `INVALID_SUMMARY_MODE` | 要約モード不正 |
| 413 | `FILE_TOO_LARGE` | サイズ超過 |
| 422 | `TEXT_EXTRACTION_FAILED` | テキスト抽出失敗 |
| 422 | `SUMMARY_GENERATION_FAILED` | 要約生成失敗 |
| 502 | `OPENAI_API_ERROR` | OpenAI API 呼び出し失敗 |
| 503 | `LOCAL_LLM_UNAVAILABLE` | ローカル LLM 利用不可 |
| 503 | `LOCAL_LLM_MODEL_MISSING` | ローカル LLM モデル未設定 |
| 503 | `LOCAL_LLM_TIMEOUT` | ローカル LLM タイムアウト |
| 503 | `NDLOCR_UNAVAILABLE` | `NDLOCR-Lite` 利用不可 |
| 503 | `NDLOCR_EXECUTION_FAILED` | `NDLOCR-Lite` 実行失敗 |
| 503 | `NDLOCR_TIMEOUT` | `NDLOCR-Lite` タイムアウト |
| 500 | `INTERNAL_SERVER_ERROR` | 想定外エラー |

## 8. Schema Design

## 8.1 正常レスポンス

```python
from pydantic import BaseModel, Field

class SummarizeResponse(BaseModel):
    filename: str = Field(..., description="Uploaded file name")
    ocrMode: str = Field(..., description="Used OCR mode")
    summaryMode: str = Field(..., description="Used summary mode")
    summary: str = Field(..., description="Generated summary")
    structuredData: dict = Field(..., description="Structured summary data")
    extractedText: str = Field(..., description="Extracted and normalized text")
    warnings: list[str] = Field(default_factory=list, description="Warnings")
```

## 8.2 エラーレスポンス

```python

class ErrorDetail(BaseModel):
    code: str
    message: str

class ErrorResponse(BaseModel):
    error: ErrorDetail
```

## 9. Internal Service Design

## 9.1 推奨レイヤ構成

- `router`
  - HTTP 入出力
  - フォーム値の受け取り
  - レスポンス組み立て
- `pipeline`
  - 処理全体のオーケストレーション
- `ocr service`
  - OCR モードごとの抽出処理
- `normalization service`
  - 抽出テキスト整形
- `summary service`
  - 要約モードごとの整形 / 要約処理

## 9.2 OCR サービス抽象化

責務:

- `ocr_mode` に応じた OCR 実装を切り替える
- 共通の抽出テキスト I/F を返す

```python

    file_bytes: bytes,
    content_type: str,

    ocr_mode: str,

    ...
```

内部実装候補:

- `extract_with_openai(...)`
- `extract_with_ollama(...)`
- `extract_with_ndlocr_lite(...)`

責務:

- 抽出テキストの整形

- warnings の生成

想定 I/F:

## 10. Local Integration Notes

## 10.1 Ollama 連携

前提:

- Ollama サーバが起動していること
- 対象のマルチモーダルモデルが取得済みであること

考慮点:

- モデルごとの応答形式差異
- タイムアウト
- ローカル実行時間の増加
- 画像入力仕様の差異
- `ocr_mode=local_llm` と `summary_mode=local_llm` は別モデルを設定可能とする
- モデル未設定時は自動補完せず、明示エラーを返す

## 10.2 `NDLOCR-Lite` 連携

前提:

- `C:\Users\ryo-n\Codex_dev\ndlocr-lite` が参照可能であること
- Python 実行環境と依存関係が整っていること

連携方式候補:

1. CLI 実行
2. Python モジュール呼び出し

初期実装では CLI 実行のほうが疎結合で扱いやすい。

Phase 2 では CLI 実行を正式仕様として扱い、内部 import は将来最適化案とする。

## 10.3 Phase 2 固定ポリシー

- 自動フォールバックは行わない
- `high_accuracy` は `NDLOCR-Lite` 主 OCR モードとして扱う
- `summary_mode=local_llm` は自然文要約を必須とし、`structuredData` は後段補完を許容する
- ローカル依存の未整備は 503 系で返し、API サーバ起動自体は阻害しない
- Ollama の推奨既定モデルは `Qwen3.6-35B-A3B` とする
- `NDLOCR-Lite` の単一画像処理は `ndlocr-lite --sourceimg <path> --output <dir>` を正式実行形式とする
- `NDLOCR-Lite` 出力は `<stem>.txt` / `<stem>.json` / `<stem>.xml` を前提とし、本アプリでは `<stem>.txt` を一次採用する

## 11. Validation Details

### 入力バリデーション

- `file` 未指定 → 400
- `filename == ""` → 400
- `content_type` が `image/jpeg` または `image/png` 以外 → 400
- `size == 0` → 400
- `size > 10MB` → 413
- `ocr_mode` が定義外 → 400
- `summary_mode` が定義外 → 400

### 出力バリデーション

- 抽出テキストが空文字のみ → 422
- 要約結果が空文字のみ → 422

## 12. Logging Policy

### INFO

- リクエスト受付
- ファイル名
- content type
- ファイルサイズ
- `ocr_mode`
- `summary_mode`
- 処理完了時間

### WARNING

- 抽出テキストが短すぎる
- 一部不明瞭箇所あり
- warnings 返却あり
- ローカル実行で応答が遅い

### ERROR

- バリデーション失敗
- OpenAI API 失敗
- Ollama 接続失敗
- `NDLOCR-Lite` 実行失敗
- 想定外例外

ログには API キー、画像生データ、機密文書本文をそのまま出力しない。

## 13. Timeout / Retry Policy

- OpenAI API 呼び出しにはタイムアウト設定を持つ
- Ollama 呼び出しにもタイムアウト設定を持つ
- `NDLOCR-Lite` 実行にもタイムアウトを設定する
- 自動再試行は 1 回までを上限とする
- 重複実行リスクが高い処理では過剰な自動再試行を避ける
- ローカル系は既定 60 秒を採用し、設定値で上書き可能とする
